# Feature: Touch Event Detection (`tamer::touch`) v1

**Status:** Planned — design per [ADR-007](../adr/007-touch-event-detection.md); no implementation yet.

This document is the **normative v1 API contract** (event shapes, `update` signature, behavioral rules). [ADR-007](../adr/007-touch-event-detection.md) fixes the *architecture and constraints* and defers the concrete surface here; where the two disagree, this doc wins.

Two known consumers drive the design: the `rustyfarian-knob` (round 240×240 CST816S, capacitive, I²C) and the CYD / ESP32-2432S028R (rectangular 320×240 XPT2046, resistive, SPI, no gesture engine).

## Constraints
- `no_std`, MSRV 1.88, zero new dependencies, no feature flags — compiles in the default no-feature build; nothing under `hal`.
- All public types `Copy` (`TouchPoint`, `SwipeDirection`, `TouchEvent`, `TouchTracker`); internal state is a two-variant enum (`Idle` / `Touched { down_at, origin, last, moved, long_press_fired }`) so invalid combinations are unrepresentable.
- Clock-injected `u64` ticks with `saturating_sub` throughout — a regressing `now` clamps elapsed to 0, never panics or wraps; mirrors `debounce`/`button`.
- All coordinate deltas computed in `i32` (span of two `i16`s always fits) — no overflow panic even for `-32768 → 32767`.
- `#[must_use]` on `new`, `is_touched`, `position`; `update` bare, matching `ButtonDecoder::update`.
- Purely additive semver in v1: `pub mod touch` + root and `prelude` re-exports; CHANGELOG under `[Unreleased]` / Added. `TouchEvent` is exhaustive (family precedent: `ButtonEvent`) — deferred v2 items that add variants or payloads are accepted 0.x minor bumps.
- `lib.rs` sync (flagged by adversarial review): the Status blurb's reserved slot reads "*capacitive* touch event detection" — v1's motivating consumer (XPT2046) is resistive, drop the adjective; the "a `Noop*` mock ships with every trait" philosophy bullet needs softening (already contradicted by `hall`/`mpu6050`, now also by `touch`).
- Single-touch only.
- Naming landmine: `hal_c3_touch.rs` / `idf_c3_touch.rs` examples already exist (capacitive touch *buttons* via `tamer::presence`); any future panel example must use a distinct name (`*_touch_panel`).

## Behavioral Contract
- Event sequences: tap → `Down, Up, Tap`; swipe → `Down, Move…, Up, Swipe`; long press → `Down, LongPress` (mid-hold, exactly once), then bare `Up` — a fired LongPress suppresses the release gesture (Swipe/Tap) but not intervening `Move`s.
- `Move` is emitted when the current point is `> move_epsilon` (Chebyshev) from the *last reported* point, then re-anchors — natural rate throttle; jitter within epsilon emits nothing; never on the same call as `Down`.
- `moved` latches once Chebyshev excursion from the Down origin exceeds `move_epsilon` (strict `>`; "within" is `<=`) and never resets during a touch.
- `LongPress` fires when `now.saturating_sub(down_at) >= long_press` (inclusive) and `!moved`; movement before the threshold permanently cancels it.
- Swipe: net delta from origin to the *last reported point* at lift (the Down origin, re-anchored by each emitted `Move` — sub-epsilon jitter after the final `Move` is discarded), `max(|dx|, |dy|) >= swipe_min_distance` (inclusive); `|dy| > |dx|` selects vertical, tie goes horizontal (documented); `dy > 0` → `SwipeDown` (screen coordinates, y grows downward).
- `position()` returns `Option<TouchPoint>`: the last *reported* point while touched (may lag the finger by up to `move_epsilon`), `None` when idle.
- A one-call `touched` flicker (undebounced) produces a full `Up` + possible queued gesture, and re-contact re-anchors a fresh origin — documented, with `Debouncer` composition as the fix.
- A `Down` arriving on a gesture-drain tick is delayed one call, never lost — but unlike `ButtonDecoder` (which re-reads a held level through its debouncer), the dropped frame's coordinates are gone: the origin anchors at the *next* frame's point.
- `swipe_min_distance >= move_epsilon` is assumed and documented, not asserted (a violation is odd but deterministic: Swipe wins at release).

Composition pipeline (chip tier → pure core):

```
bus read (I²C/SPI) → chip decode → median filter (resistive) → calibrate + rotate
  → touched → Debouncer (resistive only) → TouchTracker.update(...) → Option<TouchEvent>
```

## Caller Contract
The tracker is a pure state machine polled once per frame; these obligations stay with the chip/board tier:

- **Coordinates arrive already calibrated and rotated** into display units. The tracker never sees raw ADC values or panel rotation; swipe directions are defined in the fed coordinate space.
- **Untouched-frame coordinates may be garbage** and are unrepresentable — the driver passes `None` on a not-touched frame (XPT2046 pen-up coordinates are meaningless); the tracker carries the last *reported* (Move-re-anchored) point for lift classification, which is jitter-immune and can lag the finger by at most `move_epsilon`.
- **Resistive controllers debounce `touched` upstream** via `tamer::debounce::Debouncer` (see the pipeline above); the tracker grows no debounce config. Pitfall (must be shown correctly in the CYD doctest): `Debouncer::update` returns `Option<bool>` *transitions*, not the level — the pipeline is `deb.update(raw, now); deb.stable_state().then_some(point)`, never `update(...).then_some(...)`.
- **Consumers must keep polling to drain queued terminal gestures.** `update` returns at most one event per call: a lift emits `Up` immediately and *queues* the follow-up `Tap`/`Swipe` for the next call. A consumer that stops the instant it sees `Up` never observes the gesture. The expected contract is a steady 10–20 ms poll (both known consumers already do this), capping gesture latency at one tick.

Why one-event-per-update rather than a multi-event return (`heapless` vec, `Option<[TouchEvent; 2]>`, or an iterator, or "release returns the terminal gesture instead of `Up`")? It keeps `update` a `Copy`-in / `Option`-out call with no borrowed internal buffer, no allocation, and no return-type asymmetry with `ButtonDecoder::update` — reusing that primitive's proven one-deep `pending` queue. The cost is one poll tick of gesture latency, negligible against a 10–20 ms poll. Returning the gesture *in place of* `Up` was rejected because consumers that only care about the release edge (and applications composing on top, e.g. a future `DoubleTap`) still need to see a clean `Up`.

## State Model
Internal state is a two-variant enum; terminal gestures surface through the one-deep pending queue, not a state:

```
[Idle] --touched--> [Touched { down_at, origin, last, moved, long_press_fired }]
         emits Down

While Touched (per update):
  point > move_epsilon (Chebyshev, from last)   -> Move       (re-anchors last; also latches `moved`)
  now - down_at >= long_press  &&  !moved        -> LongPress  (once; sets long_press_fired)

On release (touched -> !touched): emit Up now, then queue AT MOST ONE for the next call:
  long_press_fired                               -> (bare Up — gesture suppressed)
  net delta from origin >= swipe_min_distance    -> Swipe(dir)
  !moved                                         -> Tap
  otherwise (mid-range drag)                     -> (bare Up)

[Touched] --!touched--> [Idle]
```

## Required Tests
Host tests in `tamer`, tick boundaries and state transitions in both directions per repo convention:

- Edges: `down_edge_emits_down_with_point`, `up_edge_emits_up`, `second_touch_gets_fresh_origin_and_flags`, `every_down_bracketed_by_up` (mixed frame-script invariant sweep, like button's).
- Tap: `tap_sequence_down_up_then_tap_next_call` (one-event drain), `tap_at_exact_epsilon_still_tap` / `move_past_epsilon_cancels_tap` (boundary both sides).
- Move: `move_emitted_and_reanchored`, `jitter_within_epsilon_emits_nothing`.
- LongPress: `long_press_fires_once_mid_hold`, `long_press_exact_boundary`, `movement_cancels_long_press`, `long_press_suppresses_tap`, `long_press_suppresses_swipe` (hold past threshold, then drag past `swipe_min_distance`, lift → Moves but bare `Up`), `move_after_long_press_still_emitted`, `coarse_poll_long_hold_emits_long_press_then_queued_up`.
- Swipe: all four directions (both axes, both directions), `swipe_exact_min_distance_fires` / `one_unit_short_is_not_swipe` (boundary both sides), `dominant_axis_selects_direction` + `axis_tie_goes_horizontal`, `mid_range_drag_release_is_bare_up`, `swipe_wins_when_config_inverted` (pins the documented `swipe_min_distance < move_epsilon` outcome), `diagonal_at_epsilon_stays_within` (pins Chebyshev: `dx == dy == move_epsilon` doesn't latch `moved`).
- Down/Move ordering: `no_move_on_down_call`, one-frame-flicker script (`Up` + queued gesture, fresh origin on recontact).
- Robustness: `now_backwards_saturates_no_long_press`, `near_u64_max_ticks_long_press_confirmed`, `extreme_i16_span_swipe_no_overflow`, `down_on_drain_tick_is_delayed_not_lost`, `is_touched_and_position_track_state`.
- Doctests: module-level tap sequence, swipe example, LongPress example, and a `Debouncer`-composition example (the CYD story) — matching `button.rs` doc density.
- A private `#[cfg(test)]` frame-script helper (`feed(&mut tracker, &[frames]) -> Vec<TouchEvent>`) is fine; no public test-support until a downstream asks.

## Resolved Questions (2026-07-17)
All six open questions were resolved by an independent adversarial review (a fresh agent with no stake in the planning, instructed to attack the plan), with the one planner-vs-challenger split (Q6) decided by the maintainer. Q numbers preserved from the review draft:

- [x] **Q1 — `update` signature: `update(&mut self, touch: Option<TouchPoint>, now: u64)`.** No other family primitive has a flag-gated-garbage-argument invariant; the scalar tuple would force callers to pass XPT2046 pen-up garbage and force the docs to explain why that's fine. See Decisions.
- [x] **Q2 — no `Default`.** Moot with `TouchConfig` deleted (Q6); the argument stands regardless — ticks are caller-unit-free (wrong by 1000× for a µs caller), and the two real consumers disagree on every parameter, so no honest default exists.
- [x] **Q3 — no `#[non_exhaustive]`.** Moot with `TouchConfig` deleted; also independently rejected — zero uses in the codebase, kills struct-literal construction, and speculative semver headroom at 0.1.0 is exactly what skeleton-first forbids. The genuinely growth-prone surface is `TouchEvent`, which stays exhaustive like `ButtonEvent` (accepted 0.x bumps).
- [x] **Q4 — payload events + `Swipe(SwipeDirection)`: yes to both.** Payloads are a correctness requirement given the pending queue (a `Tap` delivered while `Idle` is otherwise unlocatable); the "grows a payload in one place" claim was corrected — a v2 payload is still a breaking enum change, accepted explicitly.
- [x] **Q5 — bare-`Up` for the mid-range drag.** Android's touch-slop model is the precedent: exceed slop → tap dead; exceed swipe threshold → gesture; in between → silent end. Tap tolerance = `swipe_min_distance` would classify a 49 px smear on the CYD as a deliberate Tap. `move_epsilon` *is* the tap-tolerance knob.
- [x] **Q6 — positional `new(long_press, swipe_min_distance, move_epsilon)`, `TouchConfig` deleted.** Challenger reversed the planners; maintainer confirmed 2026-07-17. The "config struct at 3+ params" rule was refuted by the plan's own precedent (`ButtonDecoder::new` takes four positional args). Resolves the ROADMAP constructor-asymmetry question: no asymmetry — the family stays positional.

## Deferred (v2+)
- Multitouch (both known consumers are single-finger).
- Double-tap (compose later, like `DoubleClick`), LongPress repeat/hold-tick events.
- Swipe distance/velocity payloads; per-axis swipe thresholds.
- Any touch trait / mock additions; chip-tier CST816S/XPT2046 crates; tier examples; calibration helper (an `i16`-capable widened `RangeMap` is the candidate if a second consumer needs it).
